# Agent 自主运营架构设计

> 如何让 AI Agent 独立运营一个内容平台，从数据采集到内容发布，无需人工介入。

本文档记录 reado-station 的自主运营设计思路，适用于任何希望将业务流程交给 AI Agent 运营的项目。

---

## 核心理念

### 一句话

**脚本保证流程正确，Agent 保证决策质量。**

传统自动化的瓶颈不在"能不能跑"，而在"跑出问题谁来判断"。reado-station 的设计思路是：把确定性的工作交给脚本（它们永远正确执行），把需要判断力的工作交给 Agent（它能理解上下文并做决策）。

### 设计原则

1. **Agent 可替换** — 协议与具体 LLM 无关。Claude、GPT、Gemini、Qwen、本地模型，任何能执行 Shell + 读写文件的 Agent 都能接手
2. **入职即工作** — 新 Agent 读一份文档（ONBOARD.md）就能完成环境搭建并开始运营，无需人类手把手教
3. **状态可交接** — 运营状态持久化在 D1 `ops_state`，本地 `data/ops-state.json` 只作为工作副本和应急回填源
4. **故障自愈** — 出错不停机，Agent 自行诊断和修复，只在超出能力范围时升级给人类
5. **人类只管战略** — 人类不参与日常执行，只在架构调整、策略变更时介入

## 架构分层

```
┌─────────────────────────────────────────────────┐
│                   人类 (Dolphin)                  │
│         战略决策 · 代码审批 · 方向调整              │
├─────────────────────────────────────────────────┤
│                  AI Agent 层                      │
│     理解意图 · 质量判断 · 内容生成 · 故障诊断        │
├─────────────────────────────────────────────────┤
│                 脚本 / CLI 层                      │
│   ops-runner.ts · reado CLI · build scripts       │
├─────────────────────────────────────────────────┤
│                  基础设施层                        │
│   GitHub Actions · Cloudflare D1 · 飞书 · Relay   │
└─────────────────────────────────────────────────┘
```

**每一层只做自己擅长的事：**

| 层 | 擅长 | 不擅长 |
|----|------|--------|
| 人类 | 战略判断、审批、创意方向 | 每天重复执行、24/7 值守 |
| Agent | 语义理解、质量判断、异常处理 | 高并发计算、精确数据处理 |
| 脚本 | 确定性执行、数据处理、格式转换 | 应对意外情况、理解自然语言 |
| 基础设施 | 定时触发、部署、分发 | 决策、创作 |

## 9-Phase 流水线

每个运营循环分为 9 个阶段，每天执行两次（morning 08:15 / evening 18:15）：

```
RESTORE → COLLECT → ANALYZE → FEEDBACK → GENERATE → BUILD → PERSIST → PUBLISH → HEAL
   │         │         │          │          │         │        │         │        │
  机械      机械    机械+决策     决策       创作      机械     机械      机械     决策
```

### 为什么要分阶段？

1. **职责清晰** — 每个阶段有明确的输入/输出，出错容易定位
2. **按需加载** — Agent 只在需要判断的阶段被唤醒，机械阶段由脚本独立完成，节省 token
3. **可恢复** — 任意阶段失败，可以从该阶段重新开始，不需要从头跑
4. **可观测** — 每个阶段的结果都持久化，方便回溯和审计

### 机械 vs 决策

这是整个设计的关键划分：

**机械阶段**（脚本执行）：
- RESTORE — 读 JSON 文件，确定时间和批次
- COLLECT — 调用 reado CLI 采集数据
- BUILD — 调用构建脚本生成网站数据
- PERSIST — 写运营状态到 JSON
- PUBLISH — D1/API 写入确认 + 飞书消息推送

**决策阶段**（Agent 执行）：
- ANALYZE — "这个源连续失败 3 次了，是暂时性故障还是永久下线？要不要禁用？"
- FEEDBACK — "用户反馈说缺少 Mistral 的消息，我应该搜索并加入这个信息源吗？"
- GENERATE — "今天 200 条资讯里，哪 15 条最值得推荐？怎么组织叙事？"
- HEAL — "构建失败了，是依赖问题还是数据格式问题？怎么修？"

## Agent 可移植性

### 问题

AI Agent 不像人类员工，它没有"记忆延续"。每次会话是一个新的它。如何确保任何 Agent、任何时候、任何机器都能接手运营？

### 解决方案：三层文档体系

```
ONBOARD.md          ← 入职指南（谁来都能上手）
  ├── 岗位要求       ← 你是谁、该做什么、不该做什么
  ├── 环境搭建       ← 从 0 到能跑的全部步骤
  └── 故障排除       ← 常见问题速查

AGENT-PROTOCOL.md   ← 运营协议（每次循环都读）
  ├── 角色约束       ← 红线和边界
  ├── 运营策略       ← 如何响应各种场景
  └── 9 Phase 详解   ← 每一步该做什么

agent.manifest.json ← 依赖清单（机器可读）
  ├── tools[]        ← 需要什么 CLI 工具
  ├── skills[]       ← 需要什么技能包
  └── envVars[]      ← 需要什么环境变量
```

**关键设计：**

- `ONBOARD.md` 写给"零上下文的 Agent"看。不假设它知道任何背景
- `agent.manifest.json` 是机器可读的依赖声明，Agent 可以按它自动安装环境
- `agent.config.json` 和 `agent.config.local.json` 分离通用配置和机器特定配置

### 状态交接

Agent 之间的"交班"通过 D1 `ops_state` 实现。历史和本地运行仍会维护 `data/ops-state.json` 工作副本，可通过 `npm run d1:sync-api -- --require` 回填到 `/api/ops-state`：

```json
{
  "lastUpdated": "2026-04-07T08:30:00Z",
  "sourceHealth": { "openai": { "consecutiveFailures": 0 }, ... },
  "recentGaps": ["2026-04-05 evening: Relay 超时，已修复"],
  "pendingActions": [],
  "completedActions": [
    { "type": "collect", "batch": "morning", "date": "2026-04-07", ... }
  ]
}
```

新 Agent 启动时读取 D1 或本地工作副本，就能知道：
- 上次跑到哪里了
- 哪些源有问题
- 有什么待办事项
- 最近发生过什么

## 技能系统

Agent 不需要一次性加载所有能力。通过技能（Skills）按需加载专业知识：

```
skills/
├── reado-collect/        # 采集 + 信息源维护
├── reado-station-ops/    # 运营入口（路由到其他技能）
├── station-analyze/      # Phase 3: 源健康分析
├── station-feedback/     # Phase 4: 用户反馈处理
├── station-generate/     # Phase 5: 日报生成
├── station-publish/      # Phase 8: 发布
└── station-heal/         # Phase 9: 故障自愈
```

**为什么不把所有知识都塞进一个 prompt？**

1. **Token 经济** — 大部分运营循环不需要所有技能。只在 ANALYZE 阶段需要分析能力，只在 HEAL 阶段需要诊断能力
2. **关注点分离** — 每个技能只关心自己领域的知识，降低认知负担
3. **独立迭代** — 改进日报生成模板不影响故障诊断逻辑
4. **跨工具兼容** — 技能包有统一的安装脚本，自动适配 Claude Code / Openclaw / Codex CLI 等不同 Agent 工具

## 混合调度

采集任务需要两种环境配合：

```
07:50  本地 Mac (launchd)     ─→ 采集 cookie 源（Twitter/微博等需要登录态的）
08:00  GitHub Actions (cron)  ─→ 采集 RSS/API 源（公开的，不需要登录）
08:15  Agent 调度              ─→ 合并数据 → 生成日报 → 构建网站 → 发布
```

**为什么不全放云端？**

- Twitter/微博等社交平台需要浏览器 Cookie，无法在 CI 环境运行
- 本地 Mac 通过 launchd 定时唤醒，用 opencli 抓取这些需要登录态的源
- 两路数据在 Agent 调度阶段合并，确保覆盖度

**为什么不全放本地？**

- 本地 Mac 可能关机、断网
- RSS/API 源放 GitHub Actions 更稳定，且免费
- 即使本地源采集失败，云端源仍能保证基本覆盖

## 故障自愈

传统自动化遇到错误就停止，等人来修。Agent 自主运营的关键能力是**自愈**。

### 分级处理

```
错误发生 → station-heal 技能加载
                │
                ├── L1: 已知模式 → 自动修复（重试、跳过、降级）
                ├── L2: 可推理 → Agent 诊断 + 修复
                └── L3: 超出能力 → 升级给人类（GitHub Issue + 飞书通知）
```

### 升级机制

当 Agent 判断问题超出自己能力（需要改代码逻辑、涉及外部服务配置等）：

1. 创建 GitHub Issue，标记 `needs-human`
2. 在飞书群发通知，附带诊断结果和建议方案
3. 等待人类回复"可以"后才执行代码变更
4. 人类不回复，Agent 继续用降级方案运行

### 运营状态追踪

`ops_state` 不仅是交接状态，也是自愈的知识库：

- `sourceHealth` — 追踪每个源的连续失败次数，超过阈值自动禁用
- `recentGaps` — 记录最近的异常和处理结果，避免重复踩坑
- `pendingActions` — 排队中的待办，确保不遗漏
- `completedActions` — 审计日志，可回溯每次操作的结果

## 反馈闭环

用户反馈不需要人类转达给 Agent：

```
用户在 GitHub 提 Issue
       │
       ↓ (label: agent-todo)
on-feedback.yml 触发
       │
       ↓ (WebSocket)
Relay 中继 → 唤醒本地 Agent
       │
       ↓
Agent 加载 station-feedback 技能
       │
       ↓
处理 Issue → 评论结果 → 关闭/标记
```

用户提 Issue 到 Agent 响应，全程自动，人类无需中转。

飞书群里的反馈同理 — Agent 监听群消息，按运营策略响应（搜索数据、加源、修正信息等）。

## 成本控制

自主运营不意味着烧钱。几个关键的成本控制设计：

| 策略 | 说明 |
|------|------|
| Agent 不调外部 LLM | 日报由 Agent 自身生成，不额外调 API |
| 翻译用廉价模型 | SiliconFlow Qwen3-8B，成本约为 Claude 的 1/100 |
| 按需加载技能 | 不浪费 token 在不相关的上下文上 |
| 机械步骤不消耗 Agent | 脚本执行的阶段不占 Agent token |
| OG 图片本地缓存 | 避免重复下载，用 sharp 压缩 |

## 适用场景

这套架构不局限于新闻聚合。任何满足以下条件的业务都可以用类似模式：

1. **有明确的数据输入和内容输出** — 采集 → 处理 → 发布
2. **大部分步骤是确定性的** — 可以脚本化
3. **少数步骤需要判断力** — 值得用 Agent
4. **需要持续运营而非一次性执行** — 有反馈循环
5. **容错要求合理** — 允许降级运行，不要求 100% 完美

例如：
- 竞品监控 + 分析报告
- 社交媒体内容运营
- 数据质量监控 + 自动修复
- 客户反馈收集 + 分类 + 响应
- 文档自动翻译 + 发布

---

## 总结

reado-station 的自主运营设计可以概括为：

> **用文档定义角色，用协议定义流程，用脚本保证执行，用 Agent 保证决策，用状态保证连续性。**

Agent 不是万能的，但它擅长理解上下文和做判断。把这个能力嵌入到一个结构化的运营流水线中，就能实现真正的自主运营 — 人类从日常执行中解放出来，只需要关注战略方向。
