# 当软件有了自己的「员工」

> 传统软件需要人来运营。我们试了另一条路：让 AI Agent 当员工，7x24 自主运营一个内容平台。

---

## 一、传统软件的一生，处处需要人

一个典型的内容平台（比如新闻聚合站），从开发到上线到日常运营，生命周期大概长这样：

```
开发 → 部署 → 数据采集 → 内容编辑 → 发布 → 监控 → 用户反馈 → 修复 → 下一轮采集…
```

哪些环节需要人？**几乎全部。**

| 环节 | 人做什么 | 频率 |
|------|---------|------|
| 数据采集 | 写爬虫、维护信息源、处理反爬、修复失败的源 | 每周 |
| 内容编辑 | 筛选、排序、写摘要、翻译、配图 | **每天** |
| 发布 | 检查格式、点发布按钮、发到各渠道 | **每天** |
| 监控 | 看报错日志、检查数据质量、确认服务正常 | 每天 |
| 用户反馈 | 读反馈、判断优先级、修复、回复用户 | 随时 |
| 信息源维护 | 发现新源、评估质量、接入配置、下线失效源 | 每周 |

其中**加粗**的是每天都要做的。一个人运营一个内容平台，光是"采集→编辑→发布"这个循环，每天就要花 2-4 小时。而且不能请假 — 你一天不更新，用户就觉得这个站废了。

传统自动化（cron + 脚本）能帮你解决采集和发布，但**内容编辑**和**异常处理**它搞不定 — 脚本不会判断"今天这 200 条里哪 15 条最值得推荐"，也不会在爬虫挂了的时候自己排查原因。

这就是瓶颈：**确定性的事好自动化，需要判断力的事只能靠人。**

---

## 二、Agent 时代：软件可以雇「员工」了

大语言模型改变了一件事：**判断力也可以自动化了。**

Agent 不是更高级的脚本。脚本只能执行预设指令，遇到意外就停；Agent 能理解上下文、做决策、处理异常。

这意味着软件架构可以出现一个新角色：

```
传统架构：  人 → 脚本 → 基础设施
Agent 架构：人 → Agent → 脚本 → 基础设施
```

人类从"每天执行"变成"偶尔决策"。日常运营交给 Agent。

### reado-station 的实践

reado-station 是一个 AI 资讯聚合站。我们用这个项目验证了一件事：**Agent 可以完全接管内容平台的日常运营。**

同样是上面那张表，对比一下：

| 环节 | 传统方式 | reado-station |
|------|---------|---------------|
| 数据采集 | 人写爬虫 + cron | reado CLI + opencli 自动采集 165 个源 |
| 内容编辑 | 人选稿、写摘要 | Agent 从 200+ 条中精选 15 条，生成结构化日报 |
| 发布 | 人点按钮 | 脚本自动构建网站 + 推送飞书群 |
| 监控 | 人看日志 | Agent 每次运行自动检查源健康 |
| 用户反馈 | 人读、人回 | 用户提 GitHub Issue → Agent 自动处理并回复 |
| 信息源维护 | 人找、人配 | Agent 主动发现新源、禁用失效源 |
| 故障修复 | 人排查、人修 | Agent 自行诊断，修不了才通知人 |

人的工作量从 **每天 2-4 小时** 变成 **偶尔审批一下 Agent 拿不准的代码变更**。

## 三、怎么做到的

### 核心思路：脚本保证正确，Agent 保证质量

不是什么都交给 Agent。我们把工作分成两类：

- **机械性工作**（确定性的、有固定步骤的） → 交给脚本，它们永远正确执行
- **判断性工作**（需要理解语境、需要权衡取舍的） → 交给 Agent

```
  脚本能做的         Agent 该做的
  ──────────        ──────────
  调 API 采集数据     判断"这个源挂了要不要禁用"
  JSON 格式转换      判断"这 200 条里哪些最重要"
  git push          判断"用户反馈说缺 Mistral 消息，要不要加源"
  构建静态网站       判断"构建失败了，是什么原因，怎么修"
```

### 每天的运营流程

每天跑两轮（早 08:15 / 晚 18:15），每轮 9 个阶段：

| # | 阶段 | 谁做 | 做什么 |
|---|------|------|-------|
| 1 | RESTORE | 脚本 | 读上次运营状态，确定早班/晚班 |
| 2 | COLLECT | 脚本 | 调用采集工具，从 165 个源拉数据 |
| 3 | ANALYZE | Agent | 检查哪些源挂了、覆盖度够不够、要不要调整 |
| 4 | FEEDBACK | Agent | 看看有没有用户反馈要处理 |
| 5 | GENERATE | Agent | **核心产出** — 读原始数据，写一份结构化日报 |
| 6 | BUILD | 脚本 | 把日报变成网站能渲染的格式 |
| 7 | PERSIST | 脚本 | 保存运营状态，给下一轮/下一个 Agent 交接 |
| 8 | PUBLISH | 脚本 | 推送到 GitHub（触发网站部署）+ 发飞书群通知 |
| 9 | HEAL | Agent | 任何环节出错，Agent 自己诊断修复 |

9 个阶段里，4 个需要 Agent 判断力（ANALYZE / FEEDBACK / GENERATE / HEAL），5 个是纯脚本执行。

### 数据采集：165 个源怎么抓

采集是整个系统的基础。我们用两个工具覆盖不同类型的信息源：

**reado CLI** — 自研的采集工具，主要抓 RSS 和公开 API：
- OpenAI、Anthropic、Google DeepMind 等 AI 公司官方博客
- arXiv、Hugging Face 等学术源
- TechCrunch、The Verge、36Kr 等科技媒体
- Hacker News、GitHub Trending 等开发者社区

**[opencli](https://github.com/nicepkg/opencli)** — 社交媒体采集利器，能抓需要登录态的平台：
- Twitter/X — 通过 watchlist 追踪 AI 领域 KOL
- Reddit — r/MachineLearning、r/LocalLLaMA 等 AI 子版块
- 微博、知乎、B站 — 中文社区动态
- YouTube、Medium、Substack — 英文内容创作者
- Product Hunt、Dev.to — 产品和开发者社区

opencli 的核心能力是**模拟浏览器行为**，复用本地 Chrome 的登录态（Cookie），像真人一样浏览和提取内容。对于那些没有 RSS、限制 API、甚至有反爬的平台，opencli 是关键武器。

```
信息源分布：

  RSS / 公开 API (reado)     ████████████████████  84 个
  社交媒体 (opencli)          █████████████████     69 个
  Twitter watchlist           ██                     5 个
  Telegram                    █                      2 个
  其他                        ██                     5 个
```

所有采集都在**本地 Mac** 上完成。一台始终在线的 Mac Mini 通过 launchd 定时触发采集脚本，早晚各一次。不依赖云端 CI — cookie 源需要浏览器登录态，这些敏感信息不适合上传到云端。

### Agent 可以随时换人

这是一个关键设计：**运营协议和具体用哪个 AI 模型无关。**

Claude、GPT、Gemini、Qwen，甚至本地跑的开源模型 — 只要能执行 Shell 命令 + 读写文件，就能接手运营。

怎么做到的？三份文档：

| 文档 | 给谁看 | 作用 |
|------|-------|------|
| `ONBOARD.md` | 新 Agent | 入职指南 — 从零开始搭环境、理解岗位、开始工作 |
| `AGENT-PROTOCOL.md` | 当值 Agent | 运营手册 — 每次循环按这个跑 |
| `ops-state.json` | Agent 之间 | 交接记录 — 上次跑到哪了、什么源有问题、有什么待办 |

新 Agent 上线，读一份文档就能接手。不需要人类手把手教，也不需要和上一个 Agent"交流" — 所有状态都在文件里。

### 出了问题怎么办

传统自动化遇到错误就停，等人来修。Agent 不一样 — 它能自己诊断。

```
出错 → Agent 加载 HEAL 技能
           │
           ├── 已知问题（某个源超时、格式变了）→ 自动修复
           ├── 可推理的问题（新的报错、依赖冲突）→ Agent 分析 + 尝试修复
           └── 搞不定（需要改核心代码、外部服务挂了）→ 通知人类
```

大部分日常故障（源超时、数据格式异常、构建缓存问题）Agent 能自行解决。只有涉及核心代码变更的情况，Agent 才会在飞书群发消息，等人类审批后再动手。

### 用户反馈也是自动处理的

用户在 GitHub 提一个 Issue，整个处理流程不需要人类中转：

```
用户提 Issue → GitHub 触发通知 → Relay 中继唤醒 Agent → Agent 处理并回复
```

比如用户说"为什么没有 Mistral 的消息"，Agent 会：
1. 搜索已有数据里有没有 Mistral 相关内容
2. 联网搜索确认 Mistral 最近有没有发布新消息
3. 如果确实缺失，找到 Mistral 的 RSS 地址
4. 加入信息源配置
5. 回复用户处理结果
6. 关闭 Issue

## 四、这个模式适合什么场景

不是所有项目都需要 Agent 运营。这套架构适合的场景：

**适合：**
- 有明确的数据输入和内容输出（采集 → 处理 → 发布）
- 大部分步骤是确定性的，可以脚本化
- 少数步骤需要判断力，值得用 Agent
- 需要**持续运营**，不是一次性任务
- 容错要求合理 — 允许偶尔降级，不要求 100% 完美

**不适合：**
- 纯计算型任务（Agent 的优势是判断力，不是算力）
- 对延迟极度敏感（Agent 推理需要时间）
- 需要 100% 准确率（Agent 会犯错，需要有兜底机制）

**类似场景举例：**
- 竞品监控 + 定期分析报告
- 社交媒体内容运营（采集热点 → 生成内容 → 发布）
- 数据质量巡检 + 自动修复
- 客服工单自动分类 + 响应
- 多语言文档自动翻译 + 发布

## 五、总结

```
传统模式：人是运营者，软件是工具
Agent 模式：Agent 是运营者，人是管理者
```

reado-station 证明了一件事：**一个人 + 一个 Agent，可以运营一个每天更新的内容平台，而这个人不需要每天花时间在上面。**

核心设计就三句话：

1. **脚本保证流程正确** — 确定性的事不需要智能
2. **Agent 保证决策质量** — 需要判断力的事交给 AI
3. **状态文件保证连续性** — Agent 可以随时更换，运营不中断

这不是未来，这是现在就能跑的东西。
